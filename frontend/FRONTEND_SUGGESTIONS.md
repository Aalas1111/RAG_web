# 前端改进建议（参考）

在现有紫黑风格与心智交互方向基础上，可考虑的后续优化：

## 1. 心智交互 / 数字节食
- **禅意模式**：增加一个「专注模式」开关，开启后隐藏顶栏按钮文字、只保留图标，或收起到侧边，减少视觉干扰。
- **动静分区**：回答区若有流式输出，将动画（打字效果）限制在回答框内，不扩散到整页。
- **渐进披露**：检索模式、高级选项等可默认折叠，需要时再展开，避免首屏信息过载。

## 2. 无障碍与包容性
- **信息密度**：在设置中提供「简洁 / 标准 / 详细」展示档位，控制列表与回答区的字号、行距。
- **键盘与焦点**：保证 Tab 顺序合理，支持仅用键盘完成「选图谱 → 输入问题 → 提交」；重要操作（如删除）有明确焦点环。
- **对比度**：在 `tailwind.config.js` 中为 `violet-400` / `violet-500` 与背景做一次对比度检查，确保正文与链接满足 WCAG AA。

## 3. 动效与反馈
- **舒缓动效**：弹窗使用 200–300ms 的 ease-out，避免快速闪烁；列表项 hover 用轻微透明度或边框过渡。
- **加载状态**：构建/增量更新进度弹窗可增加简单进度条或步骤文案（如「正在处理第 3 个文件」），若后端支持可对接进度接口。
- **成功/失败**：操作完成后用轻量 toast 或行内提示代替部分 alert，减少打断感。

## 4. 数据与体验
- **剩余次数**：选择知识图谱弹窗中的「剩余 x / 每日限额」可加轻量说明（如 tooltip：「今日剩余可查询次数」）。
- **空状态**：无知识图谱、无回答时，用插画或简短引导文案替代纯文字「暂无」。
- **错误恢复**：查询失败时提供「重试」按钮，避免用户只能重新输入。

## 5. 技术可选
- **Markdown 安全**：若回答内容不可完全信任，可为 `marked` 输出接一层 DOMPurify 再做 `v-html`，降低 XSS 风险。
- **响应式**：在较小屏（如 640px 以下）将对话框与弹窗改为全宽或接近全宽，保证触控可点区域足够大。
- **PWA**：若需离线或「添加到主屏幕」，可加 manifest 与 service worker，将 RAGweb 做成可安装的 Web App。

以上为可选方向，按需分步实施即可。
